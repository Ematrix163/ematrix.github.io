原型链继承

缺陷

  1. 原型链继承的第一个问题就是，父类的引用对象的值，对子类所有实例来说是共享的。在通过原型来实现继承时，原型实际上会变成另一个类型的实例。于是，原先的实例属性也就顺理成章地变成了现在的原型属性了。

  2. 原型链的第二个问题是：在创建子类型的实例时，不能向超类型的构造函数中传递参数。实际上，应该说是没有办法在不影响所有对象实例的情况下，给超类型的构造函数传递参数，接上个例子：

问题

方法只能在父类的构造函数中创建，无法绑定到父类的原型对象上。（如果绑定到SuperType.prototype上，子类按这个方式继承是访问不到这个父类原型的，为了能让子类访问到父类的原型，只能在父类构造函数中定义相应的方法。这样每次new一个实例对象都会复制一遍方法，没有实现共用，太浪费内存了。）

组合继承

介绍

组合继承（combination inheritance），有时候也叫做伪经典继承 。是把原型链继承和借用构造函数继承结合到一起。其实基本思路是两点：

  1. 使用原型链实现对原型属性和方法的继承；
  2. 通过借用构造函数对实例属性继承

分析一下这种继承方式：

  1. 子类中通过Supertype.call方法调用了父类的构造函数，其实类似于OOP语言中的super(args)；
  2. 但是由于原型的问题，需要将子类的原型指向父类的原型对象，因此我们就用SubType.prototype = new SuperType();得以让父类原型上的方法可以被子类调用
  3. 至于为什么还要再一步修改SubType.prototype.constructor = SubType，是因为之前你SubType.prototype = new SuperType()时，子类的原型其实就是一个父类的实例了，然后这个它的构造函数就是指向父类的构造函数，理论上应该是要指向子类的构造函数的，所以这边需要手动修改一下。如果你非常不想改这个，其实正常用我个人理解是没有问题的，唯一出问题的地方在于假如你想用constructor判断一个实例是子类的实例还是父类的实例后，就会出错。

问题

这种方法缺点相对来说就小很多了，就是它调用了两次父类的构造函数：

  • 第一次：子类的构造函数中 SuperType.call(this, name);，new一个子类的时候会调用一次父类构造函数；
  • 第二次： SubType.prototype = new SuperType();，绑定子类原型到父类实例的时候，调用一次构造函数。

缺点

使用寄生式继承来为对象添加函数，会由于不能做到函数复用而降低效率；这一点与构造函数模式类似。

优化一下之前的组合继承

怎么找到完美的方法呢？来，我们先再来回顾一下组合继承这个东西：

继承的核心思想无非就是：

  • 子类可以复用父类的方法
  • 子类可以用父类构造函数初始化

有一点我们觉得可以改进的是，对于这句:

SubType.prototype = new SuperType();

你直接把子类的原型指向到一个新的父类实例上，可以是可以，但是也导致了这样多调用了一次父类构造。所以啊，能不能优化一下呢？想想我们的目的不就是让子类能访问到父类的原型吗，我们尝试这样可以不可以呢？

SubType.prototype = Super.prototype
